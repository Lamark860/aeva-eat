## A. Лента — три тюнинга (не ломать, дотюнить)

Дизайнер согласен с упрощённой 2-колоночной лентой на 70%. Три добавки — и большая часть рваного ритма из v2 вернётся без героики с абсолютным позиционированием.

### A1 · Full-width «звезда» раз в 5–7 карточек

Раз в 5–7 артефактов — большой полароид во всю ширину. Это кадр-«звезда»: жемчужина, или место с особенно сильным фото.

**Триггеры (one of):**
- `place.is_gem_place === true`
- Первое место в новом бакете (текущая неделя — первый артефакт; архивный бакет — заглавный)
- Хеш `place.id % 6 === 0` как fallback ритма (если в неделе нет ни жемчужин, ни первых артефактов)

**Технически:** перевести `.doska-week` с `display: flex` на `display: grid; grid-template-columns: repeat(2, 1fr); gap: ...`. Special-cells получают `grid-column: 1 / -1`. Один CSS-rule + проп `featured` у `<ArtifactCard>`. Внутри `featured` — полароид крупнее, текст, билетик чуть жирнее.

**Риск:** масонри-эффект перестаёт работать (grid выравнивает строки по высоте). Можно компенсировать `grid-auto-flow: dense` чтобы маленькие артефакты притягивались под full-width к свободному месту, или просто принять что после featured новая строка начинается чисто.

**Acceptance:** в ленте на 5+ артефактов всегда видна одна крупная карточка как акцент, не больше одной на «экран».

### A2 · Кружочек видео — разделитель строкой, не ячейка масонри

Сейчас видео-кружочек на ленте не рендерится вообще (только на странице места в `ReviewCard`). Дизайнер предлагает: видеосообщение — это маленькая интерлюдия между событиями, не вместо.

**Раскладка:** строка-разделитель — слева тонкая чернильная горизонтальная линия, в центре кружочек видео (`<VideoKruzhok>` 80–86px), справа продолжение линии и рукописная дата. Между двумя соседними бакетами либо между парами артефактов в одной неделе, если у одного из артефактов прикреплено видео.

**Технически:**
- Расширить `useFeed` чтобы каждое место с `video_url` (либо хотя бы один review с видео) рождало sibling-«разделитель»
- `<KruzhokDivider>` — новый компонент: full-width (`grid-column: 1 / -1` если grid из A1), не tilt-able, нейтральный
- Клик по кружочку — раскрывает видео inline или ведёт на /places/:id с anchored review

**Зависимость:** требует знать какие именно review имеют видео. Текущий /api/places не возвращает per-review video info. Минимально — флаг `has_video` на place. Шире — отдельный feed-эвент `kruzhok_added` (см. backend round 4).

### A3 · Билетик-only артефакт (отзыв без фото)

Сейчас если у места нет ни cover, ни review-фото — рендерится полароид с пастельным placeholder + caption. Это OK, но scrappy-skрэпбук тут проигрывает: пустой полароид = «забыл прицепить фото». Альтернатива — компактный билетик-only артефакт.

**Когда:** `place.image_url` пуст И у первого review нет `image_url`, но хотя бы один review с рейтингом.

**Раскладка:** маленький билетик с штампиком-городом, билетик-рейтинг во всю ширину артефакта, рукописная подпись с местом (без полароида). Размер ~60% полароида. AuthorTag в углу.

**Технически:** новый компонент `<TicketCard>` или режим `<ArtifactCard variant="ticket">`. В `Home.vue`: when no image — render TicketCard.

**Acceptance:** ритм оживает — полароиды чередуются с текстовыми артефактами.

## B. Backend round 4 — приоритет по импакту

Не «по списку `backend.md`», а по эффекту на продукт. Между «что задумано» и «что есть» сейчас лежит большая дыра, которую закрывает только бэк. Без этого приложение остаётся на 60% от задумки.

### B1 · `review_photos` (стопка фото)

**Самый дешёвый огромный визуальный апгрейд.** Одна таблица, миграция, расширение API. Лента сразу оживает в полтора раза. Делать ПЕРЕД всем остальным.

```sql
CREATE TABLE review_photos (
  id BIGSERIAL PRIMARY KEY,
  review_id BIGINT NOT NULL REFERENCES reviews(id) ON DELETE CASCADE,
  url VARCHAR(255) NOT NULL,
  position INT NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

API:
- `POST /api/places/:id/reviews/:rid/photos` (multipart, до 5 за раз)
- `DELETE /api/places/:id/reviews/:rid/photos/:pid`
- Расширить ответ review: `photos: [{id, url}]` (сохранить `image_url` как backwards-compat — это первое фото)
- Migration: `image_url` существующих review → `review_photos[0]`

Фронт: дизайнерская стопка из 2-3 полароидов с лёгкими наклонами (см. DESIGN-DECISIONS L3).

### B2 · `feed_events` + `notes`

Открывают «доску воспоминаний» как класс сущности. Сердце приложения. Без них приложение — каталог мест.

- `notes` таблица + CRUD (см. backend.md §notes)
- `feed_events` (VIEW или таблица): объединить review_added/note_added/wishlist_added/wishlist_struck/gem_found/kruzhok_added в одну хронологию
- `GET /api/feed` (с курсором), `GET /api/feed/weeks` (для свернутых полосок)
- Фронт: расширение `useFeed` на event-driven вместо place-driven
- Включает edge case L1: `(place_id, date, attendees)` группировка

### B3 · `/cities/:name` + `/users/:id`

Открывают шаринговые сценарии. «Скинь мне нашу Москву», «покажи Анины места». Эти ссылки — это причина, по которой круг будет говорить про приложение друг с другом. Виральность внутри своих.

- `GET /api/cities` (агрегаты)
- `GET /api/cities/:name`, `…/places`, `…/gems`
- `GET /api/users/:id`, `…/places`, `…/gems`, `…/cities`
- Фронт: новые view `/cities/:name`, `/people/:id`

### B4 · `/gems` hub

Приятно, но без него можно жить ещё долго. Жемчужины уже видны на главной (badge) и в Найти (полка). Делать после B1–B3.

### B5 · `/random` (серверный) + расширенные фильтры

`exclude_visited_by`, `attended_by`, `visit_from/to` — детали, не блокеры. После B1–B4.

## C. Продуктовые хвосты — на горизонт MVP+

Шесть штук в порядке важности. Эти вещи не в `STATUS.md` сейчас — это новый класс задач.

### C1 · Onboarding нового пригласённого

Аня впервые залогинилась — что она видит? Если пустой скрапбук — холодно. Дизайнер предлагает: сначала «вот что у нас уже было, посмотри», потом «теперь твоя очередь прикнопить».

**Имплементация:**
- Маркировать пользователя в БД полем `joined_at`
- На первой сессии (`auth.user.places_count === 0`) показывать «обзорный режим» Доски: read-only история круга + soft-CTA «выбери первое место чтобы прикнопить» внизу
- После первого review — режим выключается

**Зависимость:** `feed_events` (B2) либо для MVP можно сделать на текущем /api/places.

### C2 · Polling / сигнал нового

Пользователь открыл через 3 дня — есть ли подсказка «Серёжа добавил 4 места»? Сейчас нет.

**Минимум:** точка на табе Доска (есть непрочитанное).

**Имплементация:**
- На бэке поле `last_seen_feed_at` у user
- `GET /api/feed/unread-count` — считает события после last_seen
- Фронт: каждые 60s или при focus вкладки опрашивает; точка-индикатор на BottomTabBar
- При открытии Доски обновляет last_seen

**Зависимость:** нужен `feed_events` (B2). Точечный костыль до этого: считать `places.created_at > last_seen` для приближения.

### C3 · Шаринг ссылки наружу

Друг кидает в общий чат «вот этот хинкальный супер» — что увидит человек по ссылке? Сейчас — login wall.

**Дизайнерское решение:** публичная preview-страница `/places/:id/preview` (или `/p/:id`):
- Без статусов рейтингов
- Только: название, город, одно фото, штамп «наша рекомендация» (если is_gem)
- Кнопка «войти, чтобы увидеть отзывы» → /login
- OG-теги для красивого превью в мессенджерах

**Имплементация:**
- Новый эндпоинт `GET /api/places/:id/preview` (без auth) — возвращает только public-fields
- SSR или server-rendered preview-page (для OG-метатегов чтобы Telegram/WhatsApp подхватили — JS-only превью не сработает)
- Минимально: статичный HTML на `/p/:id` через nginx + бэк

**Это сильно меняет архитектуру** — текущий чисто-SPA не отдаёт серверный HTML с динамическим OG. Либо SSR (Nuxt), либо крошечный шаблонный endpoint на бэке.

### C4 · «Ваш год / сезон»

Через ~3 месяца после запуска, когда у круга накопится данных. Раз в полгода — ссылка-открытка «вот что вы съели за зиму». Повод открыть приложение людям, которые редко.

Отложено в next-next, не текущий цикл. Фиксируем на горизонт.

### C5 · Reactions / отметки на чужих визитах

«Жму руку» / «тоже хочу» / «там был» — три штампика на чужом отзыве. Не комментарии (это слишком много), а односимвольные жесты. Социалка, которая не превращает дневник в инстаграм.

Имплементация:
- Таблица `review_reactions (review_id, user_id, kind)`
- API CRUD
- Фронт: ряд штампиков на каждом review-card

Можно ждать. Не ставится в ближайшие раунды.

### C6 · Полевые тесты — обязательны после round 4

Когда раунд 4 закончится — несколько недель с живым кругом до закрытия MVP. Не для багов (баги разрабы найдут), а для проверки эмоционального тона: вызывает ли приложение желание открыть его без задачи.

Если нет — значит, дневник не получился, осталась база с красивым UI. Это самое страшное, что может случиться, и единственный способ это понять — кружок и месяц использования.

**Не код, а процесс:** запланировать 3-4 недели «бета» с 5-10 пользователями круга после round 4. Метрика — не баги, а вопрос «открываете без напоминания?».
