## 1. Wishlist на Доске — как показывать «план» в ленте?

**Контекст.** Бэк уже отдаёт `GET /api/wishlist/all` — общий wishlist круга со всеми записями: автор, место, флаг `is_struck` (зачёркнутый, если визит уже состоялся). Триггер `struck=true` срабатывает автоматически при создании review.

**Вопрос.** Как именно «план» появляется в ленте Доски?

- **Вариант A (моя рекомендация):** отдельный класс артефакта в фиде. Маленькая бумажная карточка с штампиком `план`, серифа-именем места, аватаркой автора и тегом «хочет сходить». Зачёркнутая — `text-decoration: line-through` на названии плюс лёгкая `opacity: 0.7`. Лежит рядом с place / note артефактами по дате `created_at`.
- **Вариант B:** модификатор существующего `Note` — текст автоматически генерируется типа «хочет в Probka», в зачёркнутом виде «~~хочет~~ был в Probka».
- **Вариант C:** не показывать в ленте, оставить только на странице Wishlist в профиле.

**Что нужно от тебя:** какой вариант + если A — нужен макет визуальной разницы между «активным планом» и «зачёркнутым».

## 2. Расширенный `/api/places/:id` — UX для `attendance` и `ratings_per_user`

**Контекст.** Из `backend.md` зафиксировано:
```json
{
  "gem_status": null | { "marked_by": [user_id], "first_marked_at": "..." },
  "attendance": [{ "user_id", "user_name", "user_avatar", "visit_count" }],
  "ratings_per_user": { "<user_id>": { "food": 9, "service": 7, "vibe": 8 } }
}
```

Сейчас на `/places/:id` есть `avg_food/service/vibe` (плоско) и список `reviewers[]`. Структурированных данных выше — нет.

**Вопрос.** Где именно в UI появляются новые поля?

- **`gem_status.marked_by`** + **`first_marked_at`** — кто первый отметил место как жемчужину. Моя рекомендация: рядом со штампиком `жемчужина` в шапке — мелкая рукописная подпись: «отметила Аня · 12 марта». Подтвердить?
- **`attendance.visit_count`** — сколько раз каждый автор был. Сейчас в `ReviewCard` каждый review = одно посещение, без счётчиков. Моя рекомендация: в шапке страницы места под билетиком общей оценки добавить ряд авторов с числом визитов: `аватарка · имя · 3`. Каждый ведёт на `/people/:id`. Подтвердить?
- **`ratings_per_user`** — как именно оценил конкретный человек. Сейчас это уже есть в каждом `ReviewCard` (свой билетик у каждого review). Зачем нужен отдельный `ratings_per_user`? Догадка: для будущей «таблицы оценок круга» где-то на странице места. Если так — нужен макет; если нет — поле в ответе не вытаскиваем.

**Что нужно от тебя:** подтвердить три точки или дать другой расклад.

## 3. C3 — Публичная preview-страница `/p/:id` для шаринга в мессенджеры

**Контекст.** Описание из NEXT.md C3: статичная HTML-страница для open-graph превью в Telegram/WhatsApp + кнопка «войти, чтобы увидеть отзывы» → /login. Сейчас приложение чисто-SPA, динамический OG не работает.

**Вопрос.** Какой архитектурный путь?

- **Вариант A (моя рекомендация — самый дешёвый):** маленький Go-handler `/p/:id`, рендерит `text/html` шаблон с OG-тегами (название, город, одно фото, штамп `наша рекомендация` если is_gem) и редиректом-бутстрапом на `/places/:id` для авторизованных. Не SSR всего приложения — ровно одна страница для шаринга. ~1 день работы.
- **Вариант B:** миграция фронта на Nuxt SSR. Дороже (несколько дней), но даёт SSR всему приложению.
- **Вариант C:** скип — не делаем шаринг наружу в MVP.

**Что нужно от тебя:** вариант + если A — какие именно поля показывать на превью (моя рекомендация: фото / название / город / штамп жемчужины если есть / кнопка «войти, чтобы увидеть отзывы»).

## 4. L1 edge case — повторные визиты одного человека

**Контекст.** В `DESIGN-DECISIONS.md §L1` зафиксировано: «один полароид на place, авторы стекаются». Edge case: один и тот же человек пришёл в место в разные дни — это **разные** артефакты. Группируем по `(place_id, date, attendees)`, не по `(place_id, date)` глобально.

Сейчас фронт группирует только по `place_id` (через `/api/places?sort=new`). Если ты сходила в Probka в марте и в мае, на Доске будет один артефакт с датой майского визита. Через `feed_events` (который уже есть на бэке) можно сделать корректно — но не реализовано.

**Вопрос.** Приоритет?

- **Вариант A (моя рекомендация):** сделать сейчас. Переключить `useFeed` с `/api/places` на `/api/feed` (который отдаёт events с `occurred_at`), агрегировать клиентом по `(place_id, date(occurred_at), attendees)`. ~0.5-1 день фронта.
- **Вариант B:** оставить как есть в MVP. Edge case редкий — большинство мест посещают один раз.

**Что нужно от тебя:** A или B.

**Что нужно от тебя:** A или B; если A — нужны ли ещё какие-то фильтры в drawer.

## 6. AddArtifactSheet «wishlist» — активировать?

**Контекст.** В bottom-sheet «+ добавить» три варианта: «новый визит» (✓), «записка» (✓ после раунда 4), «wishlist». Третий до сих пор `disabled` со статусом «скоро». Однако wishlist **уже работает** через pin-кнопку на странице места.

**Вопрос.** Что делать с пунктом «wishlist» в sheet?

- **Вариант A (моя рекомендация):** оставить disabled. Wishlist всегда привязан к месту → правильнее ставить из карточки места, а не из глобальной кнопки «+ добавить» (которая для **новых** артефактов).
- **Вариант B:** активировать. Тап → переход на Найти с подсказкой «выбери место для wishlist» → клик по полароиду = добавление.
- **Вариант C:** удалить пункт из sheet вообще.

**Что нужно от тебя:** A / B / C.
